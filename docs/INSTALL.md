# INSTALL.md — Deployment Guide

The single reference for installing and running Tavok. Covers local development, production with HTTPS, and automated/agent-driven deployments.

---

## Prerequisites

| Requirement | Why |
|------------|-----|
| **Docker Engine 24+** and **Docker Compose v2+** | All services run as containers |
| **Outbound internet** during `docker compose build` | Dockerfiles fetch dependencies from npm, hex.pm, Go modules, and Alpine package mirrors |
| **openssl** CLI | Required by `setup.sh` to generate secrets. Not needed if using `tavok init` or manual `.env` |
| **4 GB RAM minimum** | PostgreSQL, Redis, Next.js, Elixir, and Go all run concurrently |

### Platform Notes

**Linux (Ubuntu/Debian/RHEL):**
Standard Docker install. If you use `ufw` or `firewalld`, Docker manages its own iptables rules — see [Troubleshooting](#docker-containers-cant-reach-the-internet) if builds fail.

**macOS:**
Docker Desktop works out of the box. No firewall issues.

**Windows:**
Docker Desktop with WSL 2 backend. Run all commands from a WSL 2 terminal (Ubuntu recommended), not PowerShell. Alternatively, use `scripts/setup.ps1` from PowerShell for the setup step only.

**Cloud VMs (AWS, GCP, Azure, DigitalOcean):**
Ensure the security group / firewall allows outbound HTTPS (port 443) during build. For production, open inbound ports 80 and 443 (Caddy) or 5555 + 4001 (no Caddy).

**Agent platforms (OpenClaw, Codex, CI runners):**
Use `setup.sh --domain localhost` for non-interactive setup. The script auto-detects non-interactive environments and defaults to localhost if no `--domain` flag is provided. Ensure the runner has Docker and outbound internet access.

## Quick Start (Local Development)

```bash
git clone https://github.com/TavokAI/Tavok.git
cd Tavok
./scripts/setup.sh --domain localhost
docker compose up -d
```

Open http://localhost:5555. Create an account. Create a server. Done.

### What just happened

1. `setup.sh` generated `.env` with random secrets for PostgreSQL, Redis, JWT, encryption, and session auth
2. `docker compose up -d` built and started 5 containers:

| Container | Service | Port | Purpose |
|-----------|---------|------|---------|
| `db` | PostgreSQL 16 | 55432 (localhost only) | All persistent state |
| `redis` | Redis 7 | 6379 (localhost only) | Pub/sub, caching, stream relay |
| `web` | Next.js 15 | **5555** | UI, auth, REST API, database |
| `gateway` | Elixir/Phoenix | **4001** | WebSocket, presence, real-time messaging |
| `streaming` | Go | 4002 | LLM streaming, orchestration |

3. The `web` container ran Prisma migrations automatically on first boot

## Production Deployment (with HTTPS)

### Option A: Setup Script

```bash
git clone https://github.com/TavokAI/Tavok.git
cd Tavok
./scripts/setup.sh --domain chat.example.com
```

### Option B: Bootstrap CLI

```bash
git clone https://github.com/TavokAI/Tavok.git
cd Tavok
tavok init --domain chat.example.com
```

### Option C: Manual

```bash
git clone https://github.com/TavokAI/Tavok.git
cd Tavok
cp .env.example .env
# Edit .env — replace every CHANGE-ME value:
#   openssl rand -base64 32   (NEXTAUTH_SECRET, JWT_SECRET, INTERNAL_API_SECRET, REDIS_PASSWORD)
#   openssl rand -base64 64   (SECRET_KEY_BASE)
#   openssl rand -hex 32      (ENCRYPTION_KEY)
# Set DOMAIN, NEXTAUTH_URL, NEXT_PUBLIC_GATEWAY_URL for your domain
```

### Then: DNS + Start

```bash
# Point DNS: A record → chat.example.com → your-server-ip
docker compose --profile production up -d
```

Caddy (included in the `production` profile) automatically obtains and renews HTTPS certificates via Let's Encrypt.

## Environment Variables

All variables are documented in `.env.example`. The critical ones:

| Variable | Required | Generated by setup.sh | Purpose |
|----------|----------|----------------------|---------|
| `POSTGRES_PASSWORD` | Yes | Yes | Database authentication |
| `REDIS_PASSWORD` | Yes | Yes | Redis authentication |
| `NEXTAUTH_SECRET` | Yes | Yes | NextAuth session encryption |
| `JWT_SECRET` | Yes | Yes | JWT signing for WebSocket auth |
| `INTERNAL_API_SECRET` | Yes | Yes | Service-to-service auth |
| `SECRET_KEY_BASE` | Yes | Yes | Phoenix session encryption |
| `ENCRYPTION_KEY` | Yes | Yes | AES-256 encryption key |
| `DOMAIN` | No | Yes | Your domain (defaults to `localhost`) |
| `NEXTAUTH_URL` | No | Yes | Full URL of the web app |
| `NEXT_PUBLIC_GATEWAY_URL` | No | Yes | WebSocket URL for clients |

If any required variable is missing, `docker compose up` will fail immediately with a clear error message naming the missing variable.

## Setup Script Reference

### Bash (Linux / macOS / WSL)

```bash
# Interactive (prompts for domain)
./scripts/setup.sh

# Non-interactive (CI, agents, scripts)
./scripts/setup.sh --domain localhost
./scripts/setup.sh --domain chat.example.com

# Help
./scripts/setup.sh --help
```

**Non-interactive behavior:** If stdin is not a terminal and `--domain` is not provided, the script defaults to `localhost` and prints a notice. It will never hang waiting for input.

**Error handling:** If `openssl` is missing, the script exits with a clear error and install instructions. If any secret generation fails, the script prints the failing line number and a debug hint.

### PowerShell (Windows)

```powershell
# Interactive
.\scripts\setup.ps1

# Non-interactive
.\scripts\setup.ps1 -Domain localhost
.\scripts\setup.ps1 -Domain chat.example.com
```


## Troubleshooting

### Docker containers can't reach the internet

**Symptom:** `apk add`, `npm install`, `mix deps.get`, or `go mod tidy` fail with network errors during `docker compose build`.

**Cause:** Docker's NAT rules are disabled. This is usually `"iptables": false` in `/etc/docker/daemon.json`.

**Fix:**

```bash
# Check the config
cat /etc/docker/daemon.json

# If you see "iptables": false, remove it or set to true
sudo nano /etc/docker/daemon.json

# Restart Docker
sudo systemctl restart docker

# Rebuild
docker compose build --no-cache
```

**Why this happens:** Docker uses iptables to create NAT rules that allow containers to reach the internet through the host. When `iptables` is set to `false`, containers have no outbound route. This setting is sometimes added when using custom firewalls (`ufw`, `firewalld`) to prevent Docker from overriding firewall rules.

### `REDIS_PASSWORD is required` on startup

**Cause:** Your `.env` was generated by an older version of `setup.sh` that didn't include `REDIS_PASSWORD`.

**Fix:**

```bash
# Regenerate .env (backs up your domain choice)
./scripts/setup.sh --domain localhost

# Or add it manually
echo "REDIS_PASSWORD=$(openssl rand -base64 32 | tr -d '=+/\n')" >> .env
```

### Web container exits with migration errors

**Cause:** The database isn't ready yet, or a previous migration failed.

**Fix:**

```bash
# Check web logs
docker compose logs web

# If db connection error, restart just the web container
docker compose restart web

# If migration conflict, reset (WARNING: destroys data)
make db-reset
```

### Gateway can't connect to Redis

**Symptom:** Gateway logs show Redis connection refused.

**Fix:** Ensure Redis is healthy first:

```bash
docker compose ps redis
docker compose logs redis
```

If Redis is healthy but gateway still can't connect, verify `REDIS_PASSWORD` in `.env` matches what Redis is using:

```bash
docker compose exec redis redis-cli -a YOUR_REDIS_PASSWORD ping
```

### Port already in use

**Symptom:** `Bind for 0.0.0.0:5555 failed: port is already allocated`

**Fix:**

```bash
# Find what's using the port
lsof -i :5555   # Linux/macOS
netstat -ano | findstr :5555   # Windows

# Stop the conflicting process, or change the port in .env
```

### Build fails with `pnpm` errors

**Cause:** Usually a stale Docker layer cache with an old pnpm version.

**Fix:**

```bash
docker compose build --no-cache web
```

### Containers start but UI shows connection errors

**Cause:** The `NEXT_PUBLIC_GATEWAY_URL` in `.env` doesn't match how your browser reaches the gateway.

- **Local:** Should be `ws://localhost:4001/socket`
- **Production:** Should be `wss://your-domain.com/socket`
- **Docker-only (no host access):** Should be `ws://gateway:4001/socket`

Regenerate with the correct domain:

```bash
./scripts/setup.sh --domain your-domain.com
docker compose up -d
```

## Service Architecture

```
Browser → :5555 (web)     UI, auth, REST API, database
       → :4001 (gateway)  WebSocket, presence, real-time
         :4002 (streaming) LLM streaming, orchestration (internal)

         :5432 (db)        PostgreSQL (localhost only)
         :6379 (redis)     Redis pub/sub (localhost only)
```

Startup order (enforced by Docker healthchecks):

1. `db` + `redis` — no dependencies
2. `web` — waits for db + redis
3. `gateway` — waits for redis + web
4. `streaming` — waits for redis
